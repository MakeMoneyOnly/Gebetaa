Step 3: API Documentation
markdownDownloadCopy code## Create Project

Creates a new project for the authenticated user.

### Endpoint
`POST /api/v1/projects`

### Authentication
Requires Bearer token

### Request Body
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| name | string | Yes | Project name (1-100 chars) |
| description | string | No | Project description |
| visibility | string | No | `public` or `private` (default) |

### Example Request
\`\`\`bash
curl -X POST https://api.example.com/v1/projects \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"name": "My Project", "visibility": "private"}'
\`\`\`

### Response
\`\`\`json
{
  "success": true,
  "data": {
    "id": "proj_123",
    "name": "My Project",
    "slug": "my-project",
    "visibility": "private",
    "createdAt": "2026-01-15T10:00:00Z"
  }
}
\`\`\`

### Error Responses
| Status | Code | Description |
|--------|------|-------------|
| 400 | VALIDATION_ERROR | Invalid request body |
| 401 | UNAUTHORIZED | Missing or invalid token |
| 409 | CONFLICT | Project name already exists |

markdownDownloadCopy code# ADR-001: Database Selection

## Status
Accepted

## Context
We need to select a primary database for storing application data.

## Decision
We will use PostgreSQL as our primary database.

## Rationale
- Strong ACID compliance
- Excellent JSON support for flexible schemas
- Mature ecosystem with Prisma ORM
- Cost-effective managed options (Supabase, Neon)
- Team familiarity

## Consequences
### Positive
- Reliable data integrity
- Flexible querying capabilities
- Strong community support

### Negative
- Requires connection pooling for serverless
- More complex than SQLite for local dev

## Alternatives Considered
- **MySQL:** Similar capabilities, less JSON support
- **MongoDB:** Flexible schema, but weaker consistency
- **SQLite:** Simple but limited scaling
