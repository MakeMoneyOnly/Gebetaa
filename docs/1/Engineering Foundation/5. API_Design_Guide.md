# ገበጣ Gebeta — API Design Guide
**Version 1.0 · March 2026**

---

## API Architecture Overview

Gebeta exposes one primary API surface to all clients:

```
https://api.gebeta.app/graphql   — GraphQL Federation endpoint (all clients)
https://gebeta.app/api/webhooks/ — REST webhooks (payment providers only)
https://gebeta.app/api/health    — Health check (monitoring only)
```

All business queries and mutations go through GraphQL. Webhooks stay as REST because payment providers (Telebirr, Chapa) cannot send GraphQL. Everything else is GraphQL.

## Authentication

### Staff Authentication

All staff-facing requests (dashboard, POS, KDS) require a Supabase JWT:

```http
Authorization: Bearer {supabase_access_token}
```

The JWT is validated by Apollo Router against Supabase's JWKS endpoint. The `sub` claim maps to `auth.users.id`. The `restaurant_id` context is resolved from `restaurant_staff` inside the resolver — never embedded in the JWT (avoids stale claims if staff changes restaurant).

### Guest Authentication

Guest requests from `/[slug]` carry a session token issued after HMAC QR validation:

```http
X-Guest-Session: {guest_session_id}
```

Guest sessions have no `auth.users` entry for anonymous users. For authenticated guests (opted into loyalty), the standard Supabase JWT is used.

### POS PIN Sessions

PIN sessions are device-local and not transmitted to the GraphQL API. The waiter POS authenticates via Supabase JWT (the device's staff session) and the PIN is used only for quick staff-switching on the device.

## GraphQL Conventions

### Naming

```graphql
# Types: PascalCase
type Order { }
type MenuItem { }

# Queries: camelCase, present tense
query GetOrders { }
query GetMenuItems { }

# Mutations: camelCase, imperative
mutation CreateOrder { }
mutation UpdateOrderStatus { }
mutation MarkItemOutOfStock { }

# Subscriptions: camelCase, present progressive
subscription OnOrderStatusChanged { }
subscription OnKDSOrderUpdated { }

# Fields: camelCase
totalPrice  # never: total_price
createdAt   # never: created_at
nameAm      # Amharic name field convention
```

### Monetary Values in GraphQL

**All monetary values in GraphQL responses are in Santim (INTEGER).**  
The client is responsible for dividing by 100 for display.

```graphql
type Order {
  id:             ID!
  totalPrice:     Int!       # SANTIM — divide by 100 for ETB display
  discountAmount: Int!       # SANTIM
}

type MenuItem {
  id:    ID!
  price: Int!                # SANTIM
}

# Display convention in UI:
# (order.totalPrice / 100).toFixed(2) + ' ETB'
```

### Bilingual Fields

All user-visible text fields have both English and Amharic variants:

```graphql
type MenuItem {
  id:          ID!
  name:        String!        # English
  nameAm:      String         # Amharic (nullable — not all items may have Amharic yet)
  description: String
  descriptionAm: String
}

# Client should: nameAm ?? name (prefer Amharic, fall back to English)
```

### Pagination

All list queries use cursor-based pagination:

```graphql
type OrderConnection {
  edges:    [OrderEdge!]!
  pageInfo: PageInfo!
}

type OrderEdge {
  cursor: String!
  node:   Order!
}

type PageInfo {
  hasNextPage:     Boolean!
  hasPreviousPage: Boolean!
  startCursor:     String
  endCursor:       String
}

query GetOrders($restaurantId: ID!, $first: Int = 20, $after: String, $status: OrderStatus) {
  orders(restaurantId: $restaurantId, first: $first, after: $after, status: $status) {
    edges {
      node { id status totalPrice createdAt }
    }
    pageInfo { hasNextPage endCursor }
  }
}
```

### Error Handling

```graphql
# All mutations return a union type — never throw naked errors
type CreateOrderResult {
  success: Boolean!
  order:   Order
  error:   GebetaError
}

type GebetaError {
  code:       String!    # Machine-readable: 'INVALID_MODIFIER', 'TABLE_OCCUPIED', etc.
  message:    String!    # English message
  messageAm:  String     # Amharic message for display to staff/guests
}

mutation CreateOrder($input: CreateOrderInput!) {
  createOrder(input: $input) {
    success
    order { id orderNumber status }
    error { code message messageAm }
  }
}
```

### Error Codes

| Code | Meaning | Amharic Message |
|------|---------|----------------|
| `INVALID_MODIFIER` | Required modifier group not satisfied | ማሻሻያ ያስፈልጋል |
| `ITEM_UNAVAILABLE` | Menu item is out of stock | ምርቱ አልቋል |
| `TABLE_NOT_FOUND` | Table ID invalid for this restaurant | ጠረጴዛ አልተገኘም |
| `QR_EXPIRED` | QR code is older than 24 hours | QR ኮድ ጊዜው አልፏል |
| `INVALID_SIGNATURE` | HMAC signature verification failed | ያልተፈቀደ ተደራሽነት |
| `PAYMENT_FAILED` | Payment provider returned error | ክፍያ አልተሳካም |
| `UNAUTHORIZED` | JWT missing or invalid | ፍቃድ አለ |
| `PLAN_LIMIT` | Feature requires plan upgrade | ፕላን ያሻሽሉ |

## Subgraph Schemas

### Orders Subgraph

```graphql
# src/domains/orders/schema.graphql
extend schema @link(url: "https://specs.apollo.dev/federation/v2.0", import: ["@key"])

enum OrderStatus {
  PENDING
  CONFIRMED
  PREPARING
  READY
  SERVED
  CANCELLED
}

enum OrderType {
  DINE_IN
  TAKEAWAY
  DELIVERY
}

type Order @key(fields: "id") {
  id:               ID!
  restaurantId:     ID!
  tableId:          ID
  orderNumber:      String!
  status:           OrderStatus!
  type:             OrderType!
  totalPrice:       Int!           # SANTIM
  discountAmount:   Int!           # SANTIM
  notes:            String
  guestId:          ID
  staffId:          ID
  idempotencyKey:   String!
  items:            [OrderItem!]!
  createdAt:        String!
  updatedAt:        String!
}

type OrderItem @key(fields: "id") {
  id:             ID!
  restaurantId:   ID!
  orderId:        ID!
  menuItemId:     ID!
  quantity:       Int!
  unitPrice:      Int!     # SANTIM
  itemTotal:      Int!     # SANTIM (generated)
  modifiers:      JSON
  notes:          String
  status:         String!
  kdsStation:     String
}

type Query {
  orders(
    restaurantId: ID!
    status:       OrderStatus
    tableId:      ID
    first:        Int = 20
    after:        String
  ): OrderConnection!

  order(id: ID!): Order

  activeOrders(restaurantId: ID!): [Order!]!

  kdsOrders(restaurantId: ID!, station: String!): [Order!]!
}

type Mutation {
  createOrder(input: CreateOrderInput!):          CreateOrderResult!
  updateOrderStatus(id: ID!, status: OrderStatus!): UpdateOrderResult!
  createGuestOrder(input: CreateGuestOrderInput!): CreateOrderResult!
  cancelOrder(id: ID!, reason: String):           CancelOrderResult!
}

input CreateOrderInput {
  restaurantId:   ID!
  tableId:        ID
  type:           OrderType!
  items:          [OrderItemInput!]!
  notes:          String
  idempotencyKey: String!
}

input OrderItemInput {
  menuItemId:     ID!
  quantity:       Int!
  modifiers:      JSON
  notes:          String
  idempotencyKey: String!
}
```

## REST Endpoints (Webhook-Only)

These endpoints stay as REST permanently. Payment providers cannot send GraphQL.

### POST /api/webhooks/chapa

```typescript
// Headers required:
Chapa-Signature: sha256={hmac_signature}

// Body: Chapa webhook payload (raw JSON — read as text for HMAC verification)
{
  "event": "charge.success",
  "data": {
    "tx_ref": "gebeta-order-{orderId}",
    "amount": "450.50",
    "currency": "ETB",
    "status": "success"
  }
}

// Response (always):
HTTP 200 { "received": true }
// Return 401 only if HMAC verification fails
// Never return 4xx/5xx for business logic failures — that causes Chapa to retry
```

### POST /api/webhooks/telebirr

```typescript
// Headers required:
X-Telebirr-Signature: {signature}
X-Telebirr-Nonce:     {nonce}
X-Telebirr-Timestamp: {timestamp}

// Response: same pattern as Chapa
HTTP 200 { "received": true }
```

### GET /api/health

```typescript
// No authentication required
// Response 200 — healthy:
{
  "status": "healthy",
  "version": "1.4.2",
  "timestamp": "2026-03-06T19:00:00.000Z",
  "checks": {
    "database": "fulfilled",
    "cache": "fulfilled",
    "queue": "fulfilled"
  }
}

// Response 503 — degraded:
{
  "status": "degraded",
  "checks": {
    "database": "fulfilled",
    "cache": "rejected",    // Redis down
    "queue": "fulfilled"
  }
}
```

## Rate Limiting

Applied at Apollo Router level via Upstash:

| Scope | Limit | Window | Applies To |
|-------|-------|--------|-----------|
| Per restaurant_id | 1,000 requests | 1 second | All GraphQL |
| Per IP (unauthenticated) | 100 requests | 1 minute | Guest ordering |
| Mutation burst | 50 mutations | 10 seconds | Per restaurant |
| Analytics queries | 20 requests | 1 minute | Per restaurant |

Rate limit exceeded response:

```json
{
  "errors": [{
    "message": "Rate limit exceeded",
    "extensions": {
      "code": "RATE_LIMITED",
      "retryAfter": 1500
    }
  }]
}
```

## Third-Party Integration API (Delivery Partners)

When BEU, Deliver Addis, Zmall, or Esoora integrate with Gebeta, they receive:

### Integration Credentials

- GraphQL endpoint: `https://api.gebeta.app/graphql`
- API key scoped to their `delivery_partner_id`
- Webhook endpoint for receiving order status updates

### Available Queries (Partner Scope)

```graphql
# Delivery partners can only query:
query GetRestaurantMenu($restaurantId: ID!) {
  menuItems(restaurantId: $restaurantId, availableOnly: true) {
    id name nameAm price isAvailable
    modifierGroups { name options { name priceAdjustment } }
  }
}

# Delivery partners can only mutate:
mutation CreateDeliveryOrder($input: CreateDeliveryOrderInput!) {
  createOrder(input: $input) {
    success
    order { id orderNumber status }
  }
}

# Delivery partners receive status updates via webhook (Gebeta → Partner):
# POST {partner_webhook_url}
# Body: { event: "order.status_changed", orderId, newStatus, restaurantId }
```

### Partner API Key Scoping

```typescript
// Apollo Router resolves partner scope from API key
// Partners cannot access: staff, finances, guests, loyalty, analytics
// Partners can only create orders and read menu for their linked restaurants
```

## GraphQL Client Setup (TypeScript)

### For Next.js (Dashboard + POS)

```typescript
// src/lib/graphql/client.ts
import { GraphQLClient } from 'graphql-request';

export function createGebetaClient(accessToken: string) {
  return new GraphQLClient(process.env.NEXT_PUBLIC_GRAPHQL_URL!, {
    headers: {
      Authorization: `Bearer ${accessToken}`,
    },
  });
}
```

### GraphQL Codegen Config

```yaml
# codegen.yml
schema: https://api.gebeta.app/graphql
documents: 'src/**/*.graphql'
generates:
  src/lib/graphql/generated.ts:
    plugins:
      - typescript
      - typescript-operations
      - typescript-graphql-request
    config:
      scalars:
        JSON: 'Record<string, unknown>'
```

All GraphQL queries are in `.graphql` files co-located with the components that use them. Never write inline GraphQL strings.

## API Versioning & Deprecation

Gebeta follows GraphQL's deprecation-first approach. **Fields are never removed without a deprecation cycle.**

```graphql
type Order {
  # Deprecated: use totalPrice (santim) instead
  totalPriceEtb: Float @deprecated(reason: "Use totalPrice in santim. Will be removed 2026-09-01.")
  totalPrice:    Int!   # SANTIM — the correct field
}
```

**Deprecation process:**
1. Add new field alongside deprecated one
2. Mark old field `@deprecated` with removal date (minimum 60 days)
3. Notify all known API consumers via Telegram/email
4. Remove field after deprecation date
5. Apollo Studio breaking change detection runs in CI — blocks deploy if removal is premature
