## PART 1: CRITICAL GAPS IDENTIFIED

### 1. Observability & Monitoring (CRITICAL)

#### Gap 1.1: No Prometheus Metrics Endpoint

**Severity:** P0 Critical  
**Risk:** Grafana dashboards reference metrics that don't exist; blind spots in production

**Current State:**

- Grafana dashboard JSON exists (`docs/05-infrastructure/monitoring/grafana-dashboard.json`)
- Dashboard queries reference `http_request_duration_seconds_bucket` etc.
- No `/api/metrics/prometheus` endpoint to export these metrics
- Metrics stored in `audit_logs` table, not Prometheus-compatible format

**Impact:** Cannot monitor real-time performance; SLO violations undetected

**Action:**

1. Implement `/api/metrics/prometheus` endpoint using `prom-client` library
2. Instrument key operations: order create, payment initiate, KDS action
3. Deploy Prometheus instance or use Vercel Analytics + Sentry Performance
4. Wire Grafana to real data source

**Cost:** 3-4 days engineering, ~$50-100/month for Prometheus hosting

---

#### Gap 1.2: Alerting Without Escalation Policy

**Severity:** P1 High  
**Risk:** Critical alerts missed during off-hours; no SLA for response

**Current State:**

- Telegram-only alerting (`src/lib/monitoring/alerts.ts`)
- No PagerDuty/Opsgenie integration
- No on-call rotation or escalation
- No alert deduplication or silencing

**Impact:** Sev1 incidents may go unnoticed for hours

**Action:**

1. Integrate PagerDuty or Opsgenie for on-call management
2. Define escalation policy: 5min → secondary on-call → engineering manager
3. Add alert deduplication using Redis
4. Create maintenance window silencing

**Cost:** 2 days engineering, ~$21-50/month PagerDuty

---

### 2. Disaster Recovery (CRITICAL)

#### Gap 2.1: RTO Target Untested

**Severity:** P0 Critical  
**Risk:** 30-minute RTO claim is unvalidated

**Impact:** Actual RTO unknown; may be hours longer than claimed

**Action:**

2. Validate cold standby deployment works
3. Automate Cloudflare Pages sync
4. Document actual RTO from drill results

**Cost:** 1 day for drill, ongoing quarterly

### 3. Security Controls (HIGH PRIORITY)

#### Gap 3.1: GraphQL Security Controls Incomplete

**Severity:** P1 High  
**Risk:** DoS via expensive queries; schema exposure

**Current State:**

- GraphQL introspection should be disabled in production (documented as Sprint 5)
- Query depth limiting (`max_depth: 10`) not implemented
- Alias limiting (`max_aliases: 30`) not implemented
- Query complexity analysis not enabled

**Impact:** Vulnerable to DoS attacks via expensive nested queries

**Action:**

1. Disable introspection in production
2. Add query depth limiting to Apollo Router config
3. Implement query complexity analysis
4. Add alias count limiting

**Cost:** 2 days engineering

---

#### Gap 3.2: Staff PIN Not Hashed

**Severity:** P2 Medium  
**Risk:** PINs stored in plain text; security exposure

**Current State:**

- `src/domains/staff/service.ts` lines 100, 138, 173 have TODO comments
- PINs compared directly without bcrypt hashing
- Comment says "TODO: In production, hash the PIN code with bcrypt"

**Action:**

1. Implement bcrypt hashing for PINs
2. Create migration to hash existing PINs
3. Update verification to use bcrypt.compare

**Cost:** 1 day engineering

## CONCLUSION

Gebeta has achieved remarkable engineering maturity for a startup. The core architecture, security model, and code quality are production-ready. However, **5 critical gaps must be addressed before launch**:

1. Payment webhooks (revenue risk)
2. RTO validation (operational risk)
3. Incident procedures (operational risk)
4. Metrics endpoint (observability blind spot)
5. Alert escalation (incident response)

**Recommended Timeline:**

- Week 1-2: Address P0 critical items (11 engineering days)
- Week 3-4: Address P1 high items (6.5 engineering days)
- Week 5+: Production launch if P0 complete

**Total Pre-Launch Investment:** ~18 engineering days + $71/month infrastructure

The platform is **2-3 weeks from production-ready** with focused execution on P0 items.
